# Skill Creator

このスキルは、Claudeの能力を拡張する効果的なスキルを作成するための包括的なガイドです。

## スキルについて

**スキルとは**: モジュール化された自己完結型のパッケージで、Claudeの機能を拡張します。

**スキルが提供するもの:**
- **専門的なワークフロー**: 複数ステップの手順
- **ツール統合**: ファイル形式、API
- **ドメイン専門知識**: 会社固有の知識、スキーマ
- **バンドルされたリソース**: スクリプト、リファレンス、アセット

## コア原則

### 1. 簡潔さが鍵

コンテキストウィンドウは共有リソースです:
- Claudeは既に賢いと仮定する
- Claudeが持っていない情報のみを追加
- 冗長な説明より簡潔な例を優先

### 2. 適切な自由度を設定

- **高い自由度**: コンテキスト依存タスク用のテキストベースの指示
- **中程度の自由度**: パラメータ付きの擬似コード/スクリプト
- **低い自由度**: 脆弱な操作用の具体的なスクリプト

### 3. スキルの構造

```
skill-name/
├── SKILL.md (必須)
│   ├── YAML frontmatter (必須)
│   │   ├── name: (必須)
│   │   └── description: (必須)
│   └── Markdown本文 (必須)
└── バンドルされたリソース (オプション)
    ├── scripts/ - 実行可能コード
    ├── references/ - 必要に応じてロードされるドキュメント
    └── assets/ - 出力で使用されるファイル
```

**含めないもの:**
- README.md
- INSTALLATION_GUIDE.md
- CHANGELOG.md
- その他の補助ドキュメント

### 4. 段階的開示の設計原則

3段階のロードシステム:
1. **メタデータ** (~100語) - 常にコンテキスト内
2. **SKILL.md本文** (<5k語) - スキルがトリガーされたとき
3. **バンドルされたリソース** - Claudeが必要に応じて

## スキル作成プロセス

### ステップ1: 具体例で理解する

使用パターンが明確でない限り、このステップをスキップしないでください:
- 「このスキルはどんな機能をサポートすべきですか？」
- 「このスキルの使用例を教えてください」
- 「このスキルをトリガーする発言は何ですか？」

### ステップ2: 再利用可能なスキルコンテンツを計画

具体例を分析し、再利用可能なリソースを特定:
- スクリプト、リファレンス、アセット
- 例: PDFを回転 → `scripts/rotate_pdf.py`が必要
- 例: フロントエンドWebアプリビルダー → `assets/hello-world/`テンプレートが必要

### ステップ3: SKILL.mdを編集

**Frontmatter要件:**
- `name`: スキル名
- `description`: 主要なトリガーメカニズム
  - スキルが何をするか
  - いつ使用するか
  - すべての「使用タイミング」のコンテキスト

**本文:**
- スキルの使用方法の指示を記述
- 命令形/不定詞形を使用
- バンドルされたリソースを参照
- 約500行以下に抑える

### ステップ4: 反復改善

- 実際のタスクでスキルを使用
- 非効率性に気づく
- SKILL.mdまたはバンドルされたリソースを更新
- 再度実装してテスト

## 主要ガイドライン

1. **コンテキストウィンドウ**: メタデータ（常時）、SKILL.md本文（トリガー時）、リソース（必要時）のバランス
2. **自由度**: タスクの脆弱性に応じて具体性を調整
3. **段階的開示**: SKILL.mdを簡潔に保ち、詳細はリファレンスで
4. **簡潔性**: すべてのトークンの必要性を検討
5. **整理**: リファレンスは1階層まで、深いネストを避ける

## 日本語でのスキル作成時の追加ヒント

- 指示は日本語で明確に記述
- 例は日本語のユースケースを含める
- 用語は一貫性を保つ（例: 「コンポーネント」と「部品」を混在させない）
